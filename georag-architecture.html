This is the single source of truth for GeoRAG's architecture. When code and this document disagree, the document is correct and the code needs fixing (or this document needs an explicit, deliberate update with a reason).

04b — Coordinate System Detection
StepProcessDetail
1Parse metadataExtract EPSG codes, projection strings, datum references from file headers (GeoTIFF tags, shapefile .prj, GeoPackage gpkg_spatial_ref_sys, LAS header).
2Heuristic fallbackIf no metadata: analyze coordinate ranges (e.g., 6-digit easting + 7-digit northing → UTM).

Accuracy is reported per release against a validation corpus; no quality figures are claimed without a measured baseline. Per-workspace configurable project CRS.

04d — Format Inventory

Supported ingestion formats. V1 status is authoritative in §11b; this list inventories every format the architecture covers. New formats require main-session approval.

Format-name disambiguation: "Esri File Geodatabase" and "Geosoft GDB" share the .gdb extension but are unrelated formats. Esri's is a vector GIS container (read via GDAL/pyogrio's OpenFileGDB driver — in V1). Geosoft's is a binary geophysics channel store (proprietary SDK — Roadmap per §11b). Disambiguate at ingest by inspecting directory structure: Esri .gdb is a directory with gdb/a files; Geosoft .gdb is a single binary file.

04d-tile — Tile-Serving Layer (Martin)

Martin serves geological vector layers as Mapbox Vector Tiles directly from PostGIS SQL function sources. Martin is infrastructure — it owns no state and reads PostGIS via a dedicated read-only role.

Connection & role

Martin connects directly to PostgreSQL on port 5432 (internal Docker network), not through PgBouncer. Martin uses persistent connections and server-side features incompatible with PgBouncer transaction-mode pooling. Martin manages its own connection pool (pool_size: 20).

-- Dedicated read-only role for Martin
CREATE ROLE martin_ro LOGIN PASSWORD '${MARTIN_PG_PASSWORD}';
GRANT USAGE ON SCHEMA silver TO martin_ro;
GRANT SELECT ON silver.collars, silver.spatial_features, silver.drill_traces TO martin_ro;
GRANT EXECUTE ON FUNCTION silver.collars_by_project, silver.traces_by_project,
  silver.boundaries_by_project, silver.formations_by_project, silver.historic_workings_by_project
  TO martin_ro;

04i — Hallucination Prevention (six layers)

Citation-first is a first-class architectural concern, not a feature. Every claim traces to a source or it does not ship.

LayerBehavior
1 — Retrieval quality gateMinimum reranker top-1 score per query class. Below threshold → refusal path before the LLM is called.
2 — Typed output validationPydantic AI typed schema. Claim list must include source_chunk_id per claim or the response is rejected pre-emit.
3 — Numerical claim verificationEvery number in resolved-span output (grade, depth, tonnage, coordinates) must match a value in attached citation context within tolerance. Mismatches → reject + retry.
4 — Entity resolutionEvery named entity (project, hole, formation, QP, operator) must resolve to a Neo4j node ID present in the bound evidence set.
5 — Chunk provenanceEach citation snippet is verified against the source passage text via PAYLOAD_TEXT_SNIPPET for PG entities and against silver.document_passages.text_hash for documents.
6 — Geological constraint rulesSME-configured invariants (e.g. assay grade in the right unit, structural orientation in valid range, JORC/NI 43-101 classification ordering) checked before commit.
The refusal path is the product. A confident wrong answer is worse than an honest "I couldn't find this" — junior mining decisions ride on these outputs.

Workspace tenant isolation

workspace_id is enforced on every store query. PostgreSQL via row-level filters in the silver/gold views; Qdrant via payload index; Neo4j via label filter; SeaweedFS via bucket prefix. FastAPI never trusts the caller — the JWT-derived workspace scope is the authority.

answer_run_id · span_id · char_start · char_end · claim_text · citation_ids · support_type (direct / synthesized / comparative) · resolver_confidence

Requirements

Every feedback event must reference answer_run_id. Diagnostic records persist model/prompt/retrieval/citation versions. Retrieved chunk IDs are stored separately from UI citation IDs. Retrieval/rerank/context/citation stages must be queryable in SQL (not parsed from JSONB blobs).

Without normalised retrieval/span tables, you'll know users are unhappy but not whether to fix chunking, retrieval, reranking, citation binding, or generation.
10p-i — Passage & Citation Item Model

silver.document_passages — canonical passage store, populated at ingestion time

answer_citation_items — what an inline citation actually points to

citation_id (PK) · answer_run_id · passage_id (FK to silver.document_passages) · highlight_char_start · highlight_char_end · context_passages[] (FK array, ±2 sentences for inspector) · source_uri · source_type · source_date · author_qp · linked_node_ids (Neo4j) · created_at

answer_citation_spans.citation_ids references answer_citation_items.citation_id. answer_retrieval_items.passage_id references silver.document_passages.passage_id. The §10s evidence inspector reads exclusively from these two tables — never from raw retrieval JSONB.

Stability requirement: passage_id survives document re-ingestion as long as text_hash matches. If the passage text changes between revisions, a new passage_id is allocated and the old one is retained for citation backreferences. Citation links never silently break.
10q — Suggested Follow-up Questions

Suggestions generated in the same backend request but as a separate post-answer subtask using a reduced context (original query, final answer summary, cited entities + sources, answerability flags). Cannot reference entities outside the bound evidence set for that turn.

Rendering — 3 intent buckets, max 1 chip per bucket

Intent bucketPurpose
Drill deeperExpand on a specific entity, hole, formation, or claim in the answer
Compare / pivotLateral move — compare to another project, source, time period
Validate / inspect evidenceOpen evidence inspector, surface conflicts, view supporting passage

Show 0–3 chips. Prefer evidence-inspection chips when answerability is weak. If < 2 grounded chips clear ranking threshold, fall back to safe meta-chips (Show supporting passages, What data is missing here?, Compare cited sources) or omit the row.

Omit rules

Omit the chip row entirely if: generation fails · answerability confidence low · no novel grounded follow-up clears threshold · the answer was a refusal (§10u) · the response is already an evidence-inspection or export flow.

Outcome recorded per turn in answer_runs.suggestion_outcome — generated / omitted / failed. Clicking a chip submits it as a new user message with origin=suggested_followup + parent answer_run_id.

10r — Citation Precision Mode

Two-stage model: evidence-constrained generation (§04h, §09) followed by post-hoc claim-span resolution. Body streams first via query.token; resolved citations stream on query.citation_attached and attach progressively to final text spans in the UI.

ModeBehavior
posthoc_span_resolution (default)Body streams to completion, then spans resolved against bound evidence in a dedicated post-pass. Highest precision.
hybrid_delayed_attachmentBody streams; citations resolved in a sliding window with bounded delay. Lower latency, slightly lower precision.
inline_streaming_experimentalPer-token citation hints during streaming. Experimental — known to misattribute on long-range claims.

Configurable at workspace level + environment default. Persisted per answer run in answer_runs.citation_mode.

Latency accounting

generation_latency_ms + citation_latency_ms recorded independently in answer_runs.

Degraded behavior — partial citation failure

If body generation succeeds but citation resolution partially fails: answer body remains visible · unresolved spans marked as uncited · banner shows "Some citations could not be attached" · unresolved span count + resolver failure metadata logged to answer_runs · feedback path remains available so users can flag the uncited spans.

10t — Conflict & Freshness Handling

Mining and exploration users need to know when sources disagree, when newer filings contradict older reports, and when an answer is built on stale material. Silent precedence is unacceptable.

Conflict surfacing

ConditionUI behavior
Sources disagree on a numeric valueRender with both values + amber conflict badge. Click reveals side-by-side citations and precedence rule applied.
Newer filing contradicts older reportDefault to newer source. Show "superseded by [newer source, date]" annotation on the older citation in evidence inspector.
Resource estimate methodology differsSurface methodology (inferred / indicated / measured) on each citation. Do not silently aggregate across categories.

Freshness signaling

Each citation carries an age indicator: green (< 1 yr), amber (1–3 yr), red (> 3 yr or pre-most-recent-filing). Workspace can configure thresholds. Stale-only answers (all citations red) prepend a "Latest available evidence is N years old" banner.

Precedence rules — query-intent-sensitive, workspace-configurable

No global default precedence order. Precedence applies per query intent class (current-state / historical / methodology / regulatory / comparative); applied rule recorded in answer_runs.precedence_rule_applied. When precedence cannot be determined automatically, all conflicting sources are surfaced side-by-side with the conflict badge.

In junior mining, the difference between a 2018 inferred resource and a 2024 indicated resource is the difference between a story and a deal. The UI must never collapse them into one number without flagging it.
10u — Refusal & Uncertainty UX

The refusal path is the product (§04i). When grounding fails, the response shows the user what was searched, what was missing, and what they can do next.

ElementDetail
Refusal bannerClear, non-apologetic statement: "Insufficient evidence to answer this question from the current corpus." No hedging, no fake confidence.
What was searchedList of stores queried (Qdrant / Neo4j / PostGIS), query rewrites attempted, retrieval k, filters applied. Surfaces from answer_runs.rewritten_query + retrieval trace.
What was missingSpecific gaps: "no documents matching [entity]" / "no assays for hole [X]" / "no NI 43-101 newer than [date]" / "entity [Y] not in knowledge graph". Generated from retrieval trace, not LLM.
Suggested next actionsWiden scope (relax filters, expand date range), inspect what was retrieved, upload a new document, or contact SME. Each action is a one-click chip.
Inspect retrieved evidenceEven on refusal, show the top-k retrieved chunks with scores so the user can judge whether the corpus has anything relevant at all.
Feedback hookRefusals are feedback-eligible. §10p category "Missed evidence" tells the team the corpus has a gap.
PersistenceOn refusal, set answer_runs.abstained = true and populate answer_runs.refusal_reason with the structured reason code from the query.refusal event. SQL-queryable for refusal-rate analytics — never inferred from response text.
