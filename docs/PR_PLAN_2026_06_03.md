# PR plan — 2026-06-03 session work

The session produced ~80 sub-tasks of audit/recommendation work on the
`v2.1-baseline` branch with **135 uncommitted files**. This document
groups that work into **9 focused PRs** so reviewers can take each
architectural decision in isolation.

## Blocker before any PR can ship

`gh auth status` reports "not logged in" — I can't push or open PRs
without `gh auth login` first. Once auth is sorted, the workflow per
PR is:

```bash
git checkout -b <branch>
git add <files-for-this-pr>
git commit -m "<conventional-commit>"
git push -u origin <branch>
gh pr create --title "..." --body-file <doc>.md --base v2.1-baseline
```

Both remotes are configured (`origin` → GeoRAG-Intelligent-V2.0,
`v21` → V2.1). PR target is `v2.1-baseline` on `v21`.

## Why 9 PRs, not 1

A single 135-file PR is unreviewable. A 9-PR cycle:
- Lets reviewers approve each architectural decision separately
- Lets the audit-invariants CI run against each change in isolation,
  so a regression in (say) item D doesn't block items B or E
- Makes git blame useful — `git log path/to/file` actually tells the
  story instead of "see the 135-file mega-commit"
- Maps cleanly to the memory entries (`project_audit_and_fix_2026_06_03`,
  `project_recommendations_2026_06_03`) so future-you doesn't have to
  reconstruct the grouping

## Recommended ordering

Each PR is independent EXCEPT:
- PR #2 (audit item A workspace_user pivot) is a hard prerequisite for
  PR #8 (REC foundations) which lands typed `WorkspaceContext`
- All PRs depend on PR #1 (audit-invariants CI workflow) ideally
  shipping first so each subsequent PR is gated by the new CI

Suggested merge order: 1 → 2 → 3, 4, 5, 6, 7 in any order → 8 → 9.
