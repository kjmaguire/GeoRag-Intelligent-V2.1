### New live service — `app/services/support_cockpit/customer_response_drafting.py`

~220 lines. Pure async. Exports:
- `draft_customer_response(ticket_id, actor_user_id, pool=None)` — full draft
- `DraftOutcome` NamedTuple
- `_synthesize_response()` — template synthesis (synthetic stub)

Drafting pipeline:
1. SELECT … FOR UPDATE on the ticket
2. Pull most-recent `support_ticket_traces.trace_summary` (the
   investigation result from doc-phase 139)
3. Synthesize draft via `_synthesize_response(category, severity, summary)`
4. UPDATE `customer_visible_response` on the ticket
5. Emit `support.ticket.response_drafted` audit anchor

### Template engine (synthetic stub)

Composed of 4 paragraphs:
1. **Category-aware opening** — 6 variants (wrong_answer, failed_ingestion,
   failed_report, integration_issue, performance, other)
2. **Severity tone** — 4 variants (critical/high/medium/low) with escalation
   language
3. **Investigation summary** — quotes the top_cause_summary from doc-phase 139
   when present; falls back to "still gathering signal" when missing
4. **Uniform closing** — invites correction + signs off

Every draft carries an HTML comment marker:
`<!-- [synthetic_stub doc-phase 143] generated by template engine; swap for LLM drafter when §25.4 prompt locks. -->`

Real LLM-based drafting replaces `_synthesize_response` without changing
the surrounding orchestration.
